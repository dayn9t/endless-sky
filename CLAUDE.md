# CLAUDE.md - Claude Code 项目规则

## 重要限制

### 自动提交禁止

**禁止自动提交代码到版本控制系统。**

每次提交之前必须：
1. 明确获得用户确认
2. 用户审查待提交的变更
3. 用户确认提交信息内容
4. 用户明确授权后才可执行 `git commit` 和 `git push`

### 代码修改权限

**未经授权，不得修改任何源程序代码。**

允许修改的范围仅限于：
- ✅ 单元测试代码 (`test/` 目录下的测试文件)
- ✅ 集成测试代码
- ✅ 测试框架和测试工具
- ✅ 测试数据和Mock对象

禁止修改的范围：
- ❌ 核心库代码 (`lib/`)
- ❌ 客户端代码 (`client/`)
- ❌ 服务器代码 (`server/`)
- ❌ 脚本系统 (`scripting/`)
- ❌ AI代码 (`AI/`)
- ❌ 任何生产环境源代码

### 例外情况

如需修改非测试代码，必须：
1. 明确获得用户授权
2. 在提交信息中注明授权上下文
3. 遵循项目的代码审查流程


## 汉化方案指导思想

### 核心原则：优雅，不瞎搞

汉化工作必须通过**插件机制**实现，不得直接修改游戏原始数据文件。

### 优先级顺序

1. **插件数据覆盖**（最优先）：在 `plugins/zh_CN/data/` 中提供同名节点，利用游戏的覆盖机制替换原版文本
2. **源码扩展**（次之）：若游戏引擎本身不支持插件覆盖某类内容，应修改源码以支持该覆盖机制，而不是绕过它
3. **直接修改原始文件**（禁止）：不得直接修改 `data/`、`credits.txt` 等原始游戏文件

### 具体规则

- `data/` 目录下的文件：通过 `plugins/zh_CN/data/` 中的同名节点覆盖
- `credits.txt`：当前加载机制只支持追加，不支持覆盖。正确做法是修改 `source/MenuPanel.cpp`，让插件的 `credits.txt` 能够替换原版，而不是追加
- 遇到引擎不支持覆盖的情况：先修改引擎支持覆盖，再通过插件提供翻译内容

### 判断标准

修改方案是否"优雅"的判断标准：
- 原版游戏文件保持不变，玩家禁用插件后能完整恢复英文原版
- 翻译内容完全封装在插件目录内
- 不引入 hack 或绕过游戏既有机制的做法
